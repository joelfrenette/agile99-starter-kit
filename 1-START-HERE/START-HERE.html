GITHUB SETUP

GitHub stores every version of your code. Each V0 chat becomes a GitHub branch. When a feature is tested and approved, you merge its branch into main -- which automatically deploys to production.

► What is a branch? (read if you are new to GitHub)
Understanding branches is the key to Agile99
▼

Think of main as your live production app. A branch is a safe copy where you make changes without affecting main.

In Agile99 every V0 chat creates one branch. When a feature is tested and approved you merge that branch to main -- which deploys it to production.

main ────────────────────────────────────→ production (always live) | ↑ +── feat/login-plan Chat 1 PLAN | +── feat/login-build Chat 2 BUILD | +── feat/login-code Chat 3 CODE | +── fix/login-qa Chat 4 QA | merge → ships to prod

A Feature in Five = 5 branches, one per chat, one per phase.

SUPABASE SETUP

Supabase is your database and auth system. You need two completely separate projects -- staging for testing, production for real users. They never share data. This is non-negotiable.

► Why two Supabase projects? (important -- read this)
The most common beginner mistake is skipping this step
▼

Staging is your test environment. Break things here. Run migrations here first. Test features before real users see them.

Production is where real users live. Real data. Real accounts.

If you share one Supabase project: a bad migration deletes production data. Test accounts get real emails. Half-built features reach real users.

STAGING Supabase PRODUCTION Supabase Test data only Real user data Safe to break Never touched during development Connected to Vercel Preview Connected to Vercel Production All feature branches Only main branch

// phase 3 • ~8 minutes

VERCEL SETUP

Vercel watches your GitHub repo and auto-deploys. Feature branch pushed = staging URL updated. Merge to main = production updated. The key is setting different Supabase keys for each environment.

► How Vercel separates staging from production automatically
This is the magic that makes the two-environment system work
▼

What happens automatically after setup:

Push to any branch → Vercel builds it → Deploys to unique preview URL → Uses STAGING Supabase keys Merge branch to main → Vercel builds it → Deploys to your production URL → Uses PRODUCTION Supabase keys
  • Preview environment = staging. Every branch push gets its own URL. Safe to test on.
  • Production environment = live. Only triggers when main branch updates.

You set different environment variables for each. Vercel handles the routing automatically.

Step 08
Import your GitHub repo into Vercel
Connects Vercel to your code
▼

vercel.com → Add New → Project → Import Git Repository

Find your GitHub repo → click Import

Framework Preset: Next.js (usually auto-detected) Root Directory: ./ (leave as default) Do NOT click Deploy yet -- add environment variables first.
Step 09
Set environment variables for both environments
This is where staging and production are separated
▼

Expand Environment Variables in the Vercel import screen. Add each variable and select the correct environment.

► VERCEL ENV VARS -- add each one at correct environment
-- PRODUCTION ENVIRONMENT -- NEXT_PUBLIC_SUPABASE_URL = [PRODUCTION project URL] NEXT_PUBLIC_SUPABASE_ANON_KEY = [PRODUCTION anon key] SUPABASE_SERVICE_ROLE_KEY = [PRODUCTION service_role key] -- PREVIEW ENVIRONMENT (staging) -- NEXT_PUBLIC_SUPABASE_URL = [STAGING project URL] NEXT_PUBLIC_SUPABASE_ANON_KEY = [STAGING anon key] SUPABASE_SERVICE_ROLE_KEY = [STAGING service_role key]

Double-check which keys go to which environment. Staging keys in production (or vice versa) is a painful bug to diagnose.

Click Deploy. It may fail since there is no code yet -- that is fine. The connections are set up.

Step 10
Add your Vercel domain to Supabase allowed URLs
Prevents auth errors when your app talks to the database
▼

Do this on both Supabase projects:

Authentication → URL Configuration

Site URL: Staging: https://[your-app].vercel.app Production: https://[your-domain.com] Redirect URLs -- add all three: https://[your-app].vercel.app/** https://[your-domain.com]/** http://localhost:3000/**

> The /** wildcard covers all pages. Without this, OAuth and email confirmation redirects will fail with 400 errors.

CREATE THE THREE CORE FILES

These three files are the brain of your project. They live in your repo. V0 reads them at the start of every chat and updates them at the end. They get smarter every session.

► Why these files replace an external AI chatbot
The self-contained loop explained
▼

V0 has no memory between chats. Without a system, you spend the first 10 prompts of every chat re-explaining the project.

These three files fix that. V0 reads them at the start of every chat and instantly knows the rules, the current state, and what worked before.

AI_RULES.md The laws. Grows with every painful bug discovered. CONTEXT.md Current project state. Updated end of every chat. PROMPT-LOG.md What worked, what failed. Prompts improve every session.

After 10 chats these files know your project better than any external doc. After 50 chats they are irreplaceable.

# AI_RULES.md - Agile99 Project Rules # READ THIS BEFORE DOING ANYTHING ELSE # Version: 1.0 | Project: [YOUR APP NAME] ## Confirmation required At the start of every chat output: "AI_RULES.md OK | CONTEXT.md OK | Phase: [X] | Goal: [X]" Do not write any code before this confirmation. ## Project Name: [YOUR APP NAME] Stack: Next.js App Router, TypeScript, Tailwind, shadcn/ui DB: Supabase (PostgreSQL + RLS + Auth) Deploy: Vercel (preview = staging, main = production) Repo: github.com/[HANDLE]/[REPO] ## A Feature in Five Every feature = 5 chats = 5 GitHub branches: Chat 1 PLAN (max 20 prompts) -> feat/[name]-plan Chat 2 BUILD (max 30 prompts) -> feat/[name]-build Chat 3 CODE (max 40 prompts) -> feat/[name]-code Chat 4 QA+FIX (max 40 prompts) -> fix/[name]-qa Chat 5 PROMOTE (max 20 prompts) -> merge to main ## Non-negotiable rules - All API routes must use withErrorHandler() wrapper - All protected routes must use requireAuth() middleware - Never write to DB from client-side code - Never hardcode secrets -- use environment variables only - No console.log in production code - Read CONTEXT.md before writing any code - Output only modified files -- never the full project - State what you will NOT touch before coding [ADD YOUR OWN RULES AS YOU DISCOVER THEM] ## End of every chat 1. Output full updated CONTEXT.md 2. Note what worked/failed -> PROMPT-LOG.md 3. Propose new rules for AI_RULES.md if anything broke 4. State exact starting point for next chat ## Rule change log [DATE] v1.0 - Initial rules created

YOUR FIRST FEATURE IN FIVE

Walk through a complete feature. Five chats. Five branches. One shipped, tested, deployed feature. This is your template for every feature from here on.

► The 3-minute rule -- most common beginner mistake in QA
Read this before testing on staging for the first time
▼

Every time you push code to GitHub, Vercel detects it and starts deploying. This takes 2-4 minutes.

If you open the staging URL immediately after pushing, you are testing the old version -- not your new code. This causes false bug reports and wastes significant time.

The rule: After every push, wait at least 3 minutes. Check the Vercel dashboard for the green checkmark before opening the staging URL.
